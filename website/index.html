GitReins PoC

Git-Native Agent Co-Harness — Quality Gates for AI-Generated Code

Overview

GitReins is a co-harness that lives inside your git repository. It provides MCP tools for task lifecycle management, an agentic evaluator that judges code completeness against task definitions, and git hooks that ensure nothing bypasses the quality gates.

Primary Agent

Any AI coding agent (Pi, Claude, Hermes, Codex) does the creative work — writes code, runs tests, iterates.

GitReins Co-Harness

Manages task lifecycle, enforces quality, validates completeness. The TODO items are the guardrails.

Agentic Evaluator

Not a single LLM call — an agentic loop with 5 tools that reads, tests, searches, and verifies every task criterion.

System Architecture

Agentic Evaluator — Not a Single LLM Call

The evaluator is an agentic loop with 5 purpose-built tools. It doesn't just receive a prompt and return a verdict — it reads files, runs tests, searches for patterns, and digs deeper when evidence is unclear. The loop continues until the LLM has enough evidence to deliver a verdict.

Pluggable MCP + Virtual FS Sandbox

mcp_call — External Truth Sources

The evaluator can fetch external context to verify work against real-world sources. Nothing is available by default — repos opt in via allowlist. 

# .gitreins.yaml
evaluator:
  mcp_allowlist:
    - server: jira
      tools: ["get_issue", "search_issues"]
    - server: internal_docs
      tools: ["fetch_page", "search_docs"]
    - server: database
      tools: ["query_schema"]

"Task says 'matches Jira PROJ-423' → evaluator fetches the ticket and compares."

sandbox_write/read — Evaluator Scratch Space

The evaluator gets a temporary virtual filesystem that is isolated from the repository. Used for tracking evidence, building checklists, and offloading intermediate results.

Sandbox PropertyBehavior
Location/tmp/gitreins-eval-<uuid>/
IsolationCannot touch repository code
LifespanCreated per evaluation, cleaned after verdict
PersistenceAvailable across all tool calls in one session
Use caseEvidence log, checklist, partial findings
Why this matters: Without the sandbox, the evaluator keeps everything in LLM context — expensive and lossy over a long agentic loop. With MCP, the evaluator's reach extends beyond the repo to any external truth source (Jira, docs, APIs, databases). Both are opt-in per repo via .gitreins.yaml.

Build vs Borrow — Agent Runtime Decision

We researched three approaches for the evaluator's agentic runtime. Pi's compiled binary path has known cross-platform issues (Bun 1.3.12 SIGKILL on macOS). mcp-agent (8.3K stars) and smolagents (27K stars) are full frameworks — overkill for a focused evaluator.

A: Pi (Compiled Binary)

  • 28K stars, battle-tested agent loop
  • MCP via extensions
  • Can compile to standalone binary
  • Bun build --compile has macOS SIGKILL regression
  • ~60MB binary (embeds JS runtime)
  • Requires Bun to BUILD
  • Carries full coding agent (TUI, themes, sessions)
  • Not reliable for embedded evaluator

B: mcp SDK + Custom Loop ✓

  • ~100 line custom agentic loop
  • Official mcp Python SDK for MCP protocol
  • requests for LLM API calls
  • 3 pip deps: mcp, pyyaml, requests
  • Python 3.10+ (Ubuntu 22.04 LTS standard)
  • Full control, minimal dependencies
  • No framework lifecycle — embed anywhere
  • Simple to maintain, easy to debug

C: mcp-agent / smolagents

  • mcp-agent: 8.3K stars, Apache 2.0
  • smolagents: 27K stars, HuggingFace
  • MCP-native, evaluator pattern built-in
  • Heavy async framework (MCPApp lifecycle)
  • Designed for standalone apps, not embedding
  • smolagents uses code-gen paradigm (wrong fit)
  • Significant dependency weight
  • Overkill for a ~100 line evaluator loop

Why the Agentic Loop Is Simple

while True:
    response = llm.chat(messages, tools)
    if not response.tool_calls:
        return response.content  # Verdict — agent decided it's done
    for call in response.tool_calls:
        result = execute_tool(call)  # read_file, mcp_call, sandbox_write...
        messages.append(tool_result(result))
    # Loop — LLM gets results, may call more tools

The loop itself is 10 lines. The complexity was always in MCP protocol handling — solved by the official mcp Python SDK. The evaluator calls mcp_call(server, tool, args) and the SDK handles JSON-RPC, connection lifecycle, and tool discovery.

Sandbox: gitreins Branch Persistence

The evaluator's scratch space lives on the gitreins branch, not /tmp. Every evaluation's evidence log and checklist is version-controlled, auditable, and cloneable.

Isolation

Sandbox is .git/gitreins-sandbox/ — gitignored, never touches the working tree. Evaluator writes freely here.

Persistence

After verdict: engine auto-commits sandbox to gitreins branch under history/<date>/<hash>/. Full audit trail.

Learning

Past evaluations are queryable. Agent can learn from common failure patterns across the evaluation history.

Pi Research Summary

Pi Coding Agent by Mario Zechner (pi.dev) is a TypeScript monorepo with a minimal coding agent CLI. We analyzed its architecture to inform GitReins' design:

4 Built-in Tools

read, write, edit, bash. System prompt under 1,000 tokens. Maximizes context window for actual code.

Agent Loop

Loops until the model stops calling tools. No max-steps, no iteration limits. Extension system for custom tools.

Monorepo Layers

pi-ai (LLM API) → pi-agent-core (loop) → pi-coding-agent (CLI) → pi-tui (terminal). Clean separation.

4 Modes

Interactive (TUI), Print, JSON stream, RPC (JSONL stdio). Embeddable in other systems.

Binary Compilation

Can compile to standalone binary via bun build --compile. Removes Node.js runtime requirement.

Why Not Pi for Evaluator

Pi is a general coding agent. The evaluator needs a focused judgment agent. Different purpose, different constraints.

Success Criteria

1. Instant Activation

Clone → ./gitreins/install → hooks active. Under 10 seconds.

2. Secrets Caught Instantly

Tier 1 static guard blocks secrets at pre-commit. Under 100ms.

3. Fast Evaluation

Evaluator verdict returned in under 3 seconds for typical tasks. Haiku/GPT-4o-mini.

4. Agent Workflow Works

Create task → complete items → commit. End-to-end with any MCP-compatible agent.

5. Zero Infrastructure

Everything lives in the repo. No servers, no CI, no external services beyond LLM API.

6. Universal Runtime

Python 3.9+ — works on any system with git + Python. Linux, macOS, WSL, CI runners.
